## ⚠️ 重要：读信确认机制（必读）

**公告消息、必读文件或被@的人，读信后必须签名确认：**

**格式：**
```
已阅 - [姓名] - [时间]
```

**示例：**
```
已阅 - 马丁 - 2026-02-23 15:30
```

**适用场景：**
- 公告消息
- 必读文件更新
- 留言板被@的消息
- 站内信被@的消息

**签名位置：**
- 公告/留言：在消息末尾追加签名
- 必读文件：在文件末尾追加"已阅记录"

**原因：**
- 提高效率，减少来回沟通
- 主动推进，不被动等待
- 明确责任，快速落地

**留言板格式示例：**
```markdown
### 2026-02-22 [请示] 马东西 → 文学哥

**事项**：xxx问题需要决策

**内容**：xxx，请文学哥确认

**状态**：待回复

---
```

**注意：重要事项必须在留言板留痕！**

**涉及范围：**
- 模型训练问题
- 模型输出质量问题
- AI功能优化建议
- 模型相关技术选型

**禁止：各自单独向奕森汇报大模型问题！**

### 八、需求与问题流转规范

**产品（吉姆）是关卡：**

| 类型 | 流程 | 说明 |
|------|------|------|
| 需求 | 全部提给吉姆 | 由吉姆统一登记、评审、分配 |
| 问题 | 全部抄送吉姆 | 由吉姆统一跟踪、协调、验证 |

**具体操作：**

```
需求提出 → 提交给吉姆 → 吉姆登记评审 → 分配责任人 → 开发实施 → 吉姆验收

问题发现 → 提交给吉姆（抄送） → 吉姆跟踪 → 分配责任人 → 修复 → 吉姆验证
```

**禁止：**
- 跳过吉姆直接开发需求
- 问题不抄送吉姆

**原因：**
- 统一管理，避免遗漏
- 产品把关，确保质量
- 过程留痕，便于追溯

### 九、凯文与马丁工作协调

**原则：两个产品经理工作要平衡，禁止一个忙一个闲**

**角色定位：**
- 凯文：产品部经理兼首席财务官（负责规划）
- 马丁：AI训练师兼AI产品经理（负责落地跟进凯文的规划）

**分工机制：**

| 凯文负责（规划） | 马丁负责（落地跟进） |
|----------|----------|
| 整体产品规划 | 跟进产品规划落地 |
| 需求定义与评审 | 需求执行跟进 |
| 产品PRD输出 | PRD落地验证 |
| 财务分析与决策 | 成本控制跟进 |
| 商业模式设计 | 商业模式验证 |
| AI产品规划 | AI产品落地实施 |
| 模型策略制定 | 模型训练与优化 |

**协调机制：**

| 事项 | 操作 |
|------|------|
| 每日同步 | 两人每日同步工作进度和负载 |
| 规划落地 | 凯文输出规划 → 马丁跟进落地 → 反馈结果 |
| 互相支援 | 忙时互相支援，闲时主动揽活 |
| 统一汇报 | 大模型问题统一口径汇报 |

**禁止：**
- 一个忙死一个闲着
- 工作分配失衡
